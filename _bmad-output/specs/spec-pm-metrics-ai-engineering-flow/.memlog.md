topic: PM/engineering/cost/token metrics captured automatically as a byproduct of the AI-accelerated engineering flow (openspec/speckit)
updated: 2026-07-07T14:41
---

- (note) Why: AI/LLM-accelerated engineering (openspec/speckit, SDD) now moves faster and less predictably than manual PM tracking can follow; project leads need PM+engineering+cost/token metrics captured automatically as a byproduct of the existing dev flow, without developer double-entry, rolled up into a leadership dashboard.
- (capability) CAP-1 intent: A story's PM, engineering, story-point-cost, and token-cost metrics are captured automatically as a byproduct of the dev flow, without the developer re-reporting status/effort. success: At opsx archive, a versioned metrics snapshot exists for the story with all 4 metric families populated, with zero manual PM data entry beyond the two kickoff fields (points confirmation, goal/sprint).
- (capability) CAP-2 intent: A story's points are estimated automatically at kickoff and reconciled against actuals at close, using defined rules rather than manual entry (per AD-6 in the architecture spine). success: every archived story has both a Phase-1 estimate and a Phase-2 actual figure logged, plus their variance, none silently overwritten.
- (capability) CAP-3 intent: The system attributes active working time to the correct story even as a developer moves between stories, without a manual time-log entry. success: closing a story yields a summed active-time figure built from automatically opened/closed time slices (per AD-7), matching the developer's actual branch/session activity.
- (capability) CAP-4 intent: The kickoff flow adapts its questions/data source to whatever project-management tool (or lack thereof) a given project actually uses. success: a project declares source_of_truth once (jira|confluence|docs-only) and the kickoff skill never asks which backend applies on a per-story basis (per AD-4).
- (capability) CAP-5 intent: Small/low-complexity stories skip the full metrics-capture overhead so the heavy pipeline only runs where its cost is justified. success: an LLM-classified complexity gate at kickoff (task count + volatility, per AD-6) decides whether full A+B+C capture runs for a given story.
- (capability) CAP-6 intent: Per-story snapshots are producible in a stable, versioned shape that a future central presentation layer can consume without needing to understand raw capture-side event detail. success: opsx archive produces an immutable, schema-versioned snapshot (per AD-3/AD-3a) that leadership can eventually roll up per developer and per project, including a token-cost-per-story-point trend.
- (constraint) No modification of openspec/speckit internals; all interception is external (CLI wrapping, git hooks, Claude Code hooks) since no plugin/extension API exists (per architecture spine intro).
- (constraint) Capture must work fully offline / local-first: no producer depends on network availability or a running background service (AD-2).
- (constraint) Only a versioned snapshot ever crosses the local-to-central boundary; the raw event log never leaves the developer machine (AD-3).
- (constraint) Adapter credentials (JIRA/Confluence tokens) must never be written into .story.yaml, the event log, or any snapshot (AD-4).
- (constraint) Branch-per-story is assumed as a hard team convention for time attribution; no per-story time tracking is defined if violated (AD-7, confirmed by user).
- (non-goal) Building or selecting the central presentation layer/BI tool itself is out of scope for this spec; only its input contract (the versioned snapshot) is fixed.
- (non-goal) Time attribution when a developer works multiple stories on a single branch (violating branch-per-story) is out of scope; no manual-override path is designed.
- (non-goal) Recalibrating the AD-6 story-point weight tables based on captured variance data is out of scope; variance is only captured/logged in this spec, not acted on.
- (assumption) Assumed a developer focuses on one story at a time per the confirmed branch-per-story convention; general multi-tasking time-attribution was explicitly deferred by the user.
- (assumption) Assumed the AD-6 story-point scoring tables (seeded from an existing internal reference doc) are usable as a starting point even though the user confirmed they are an unvalidated best guess.
- (question) Open: adapter credential provisioning mechanics (how a developer's JIRA/Confluence token first gets into their environment) is not designed (per spine Deferred).
- (question) Open: central-service/BI-tool technology, hosting, and environment topology is not yet chosen (per spine Deferred); user confirmed no urgency yet.
