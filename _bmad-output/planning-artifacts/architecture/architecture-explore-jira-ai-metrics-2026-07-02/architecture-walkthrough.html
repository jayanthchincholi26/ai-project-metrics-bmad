Architecture Walkthrough

Project Metrics That Track Themselves

While a developer works normally with AI tools (git, Claude Code, openspec/speckit), the system quietly captures PM, engineering, cost, and AI-token metrics in the background — no extra typing, no status meetings, rolled up into a leadership dashboard.
The problem today

AI-accelerated engineering moves faster than PM tooling can track. Developers are asked to manually re-enter what already happened — points, time, review cycles — duplicating work that the tools already know.

→
The shift

Every source (git, Claude Code, the openspec/speckit CLI) already emits the signal we need. We just have to catch it — as an event — instead of asking a human to type it in again.

The whole flow, in one line
1) Developer starts a story
→
2) Codes with AI as usual
→
3) Hooks quietly log events
→
4) Developer closes the story
→
5) A metrics snapshot is created
→
6) Dashboard updates itself

The Paradigm

Event-Sourced Pipes & Filters

Three independent sources only ever append events. Nothing writes shared state directly. A single filter — the snapshot assembler — reduces the log into one versioned snapshot at story close. Click a node below.

Invariant Group 1

Event Integrity & Namespacing AD-1, AD-1a, AD-1b

Prevents: three independent writers (git, Claude Code, the CLI wrapper) racing to edit the same file and corrupting it — or two sources emitting a same-named event with different shapes and confusing the reducer.
Why it matters: nobody has to babysit file locks or worry "did my commit hook clobber the Claude Code hook's write?" Every source just appends its own line and walks away — the log is inherently safe to write to concurrently.
{"story_id":"AIM-142","source":"git","type":"git.commit","timestamp":"...","payload":{...}} {"story_id":"AIM-142","source":"claude","type":"claude.tool_use","timestamp":"...","payload":{...}} {"story_id":"AIM-142","source":"opsx","type":"opsx.archive","timestamp":"...","payload":{...}}

AD-1 — One writer per line, every time

Each source writes its own event as a single, complete line and nothing else touches the file at that moment — like everyone dropping a note in a shared box one at a time, instead of three people trying to write on the same sheet of paper simultaneously. Corruption becomes structurally impossible, not just unlikely.

AD-1a — Namespaced types

git.*, claude.*, opsx.* — so a bare "commit" from two different sources can never collide and be misread.

AD-1b — Early events kept

An event firing before .story.yaml exists goes to a pending buffer, never dropped — the assembler backfills the story ID once the manifest lands.

TakeawayThree different tools can log events at the exact same moment without ever corrupting each other's data or getting confused about whose event is whose.
Invariant Group 2

The Snapshot Contract & Immutability AD-3, AD-3a

Prevents: the central layer needing to understand three different raw event shapes; and a re-close silently overwriting history a trend chart already depends on.
Why it matters: leadership dashboards only ever see one clean, stable shape. Re-opening or re-closing a story never erases the past — it adds a new revision, so "how did our estimate drift" can always be reconstructed.

The raw event log never leaves the developer's machine. Only this envelope crosses the boundary:

{ "schema_version": "v1", "story_id": "AIM-142", "revision": 2, // increments, never overwrites "pm_metrics": {...}, "engineering_metrics": {...}, "story_point_cost": { "phase1_points": 5, "phase2_points": 8, "variance": 3 }, "token_cost": {...} }

Every opsx archive on the same story produces a new immutable revision — nothing is ever overwritten in place.

TakeawayThe dashboard only ever has to understand one simple, stable shape of data — and history is never silently erased if a story reopens.
Invariant Group 3

Source-of-Truth Adapter Pattern AD-4

Prevents: hard-coding "everyone uses JIRA" — and breaking on the project that uses Confluence, or nothing at all.
Why it matters: one project declares its tool once; every story kickoff afterward "just works" without re-asking. Credentials also never end up inside a file that could get committed or shared.

One normalized shape, three backends

interface Adapter { points, goal, sprint, description }
jira confluence docs-only

Declared once in project config (next to project-context.md) as source_of_truth: jira | confluence | docs-only. Credentials are read from env vars / the existing credential store at call time — never written to .story.yaml, the event log, or any snapshot.

Example: Project A uses JIRA. At kickoff, the developer types a JIRA issue key (e.g. AIM-142) and the adapter fetches its points/sprint/goal automatically from JIRA. Project B has no PM tool at all — the docs-only adapter just asks the developer directly for the same four fields. Both projects hand the kickoff skill the exact same shape of data, so everything downstream (hooks, snapshot, dashboard) works identically either way, with zero code changes.
TakeawayOne project can plug in JIRA, another Confluence, another nothing at all — the rest of the system never has to know or care which.
Invariant Group 4

Story Identity Lives in the Manifest AD-5

Prevents: guessing the story ID from a branch name or ticket-key convention — conventions that not every team or repo actually follows consistently.
Why it matters: one unambiguous source of truth for "which story is this event about," so metrics never get silently attributed to the wrong story because someone named a branch unusually.

.story.yaml — written once, at kickoff

story_id: AIM-142 source_of_truth: jira points_phase1: 5 goal: "..." sprint: "2026-S14"

Every producer (git hooks, Claude Code hooks, the CLI wrapper) reads the story ID from here — never parses it out of a branch name or ticket key.

TakeawayThere's exactly one place every tool looks to answer "which story is this," so metrics can never get silently attributed to the wrong story.
Invariant Group 5

Two-Phase, Rule-Driven Story Points AD-6

Prevents: manual point entry (meaningless once code is largely LLM-generated) and silent re-estimation that hides how far off the original guess actually was.

PHASE 1  At kickoff — from openspec state alone

Task count (tasks.md)Base points
1–51–2
6–153–5
16–308–13
30+20+ — must-split

Plus a volatility bonus from openspec stage maturity: explore-only = +5 down to proposal + specs + design = 0.
Times a novelty modifier from pattern-matching prior .story.yaml records: first-time ×1.5 / standard ×1.0 / existing-pattern-reuse ×0.8.

PHASE 2  At opsx archive — from the event log

Testing typeWeight
Unit×0.5
Integration×1
Manual QA×1.5
Performance×2

Plus human review cycles (follow-up prompts / PR review rounds), agent-narrated decision events, and context management (files/components touched).

Why it matters: Phase 1 and Phase 2 are both kept, side by side, forever — the variance between "what we guessed" and "what it took" is the actual signal leadership wants, and it's never collapsed into one overwritten number.
TakeawayAn estimate is made before code exists, a real number is measured after, and the gap between them — not either number alone — is what tells leadership if estimation is improving.
Invariant Group 6

Time-on-Task via an Active-Story Pointer AD-7

Prevents: guessing time-on-task from file-edit timestamps alone — unreliable, and it mis-attributes time whenever a developer switches context.
Why it matters: one explicit pointer file says "this is the story I'm on right now," updated automatically — never a manual timer a developer forgets to start or stop.

Relies on the team's confirmed branch-per-story convention (one branch = one story = one active context). No manual override exists yet for working two stories on one branch.

Precedence rule (the tricky bit)

If a git checkout happens while a Claude Code session is already live, the live session's boundaries govern the time-slice accounting. The checkout re-points which story current activity counts toward — it does not itself open/close a session-level slice. Only SessionStart / SessionEnd do that.

This stops double-counting or truncating time when both signals land close together.

Session A (Claude Code live) Session B ● git checkout events — idle timeout auto-pauses after ~15min inactivity
Example: Priya checks out the branch for Story A at 9:00 — the pointer opens Story A's time slice. She works with Claude Code until 11:00, then a high-priority bug lands: she checks out Story B's branch mid-session. The pointer closes Story A's slice (2h logged) and opens Story B's. At 11:45 she goes to a meeting; after 15 minutes of no activity, Story B's slice auto-pauses. At 1:00 she checks Story A back out and resumes — a new slice opens for Story A. At close, Story A shows 2h + however long the afternoon resume lasted, summed automatically, with no timer she had to remember to start or stop.
TakeawayTime is tracked by watching what a developer is actually doing (checkouts, active AI sessions), not by asking them to log hours.
Invariant Group 7

Git-Versioned Hook Installation AD-8

Prevents: every developer's machine silently drifting to a different (or missing) set of hooks — since neither .git/hooks/ nor local Claude settings are version-controlled by default.
Why it matters: the whole pipeline depends on hooks actually firing. If half the team's machines are missing a hook, half the metrics quietly go missing too — this makes install a single, repeatable, committed step.

One command, once per clone

tools/setup-hooks → installs tools/hooks/git/* into .git/hooks/ → merges hook entries into .claude/settings.json

Hook logic lives in git (tools/hooks/) — only the install step is per-machine, and it's a required onboarding step, not hand-maintained.

Example: A new developer joins the team and clones the repo. Without this rule, they'd start committing and coding with none of their activity captured — a silent gap in the metrics nobody would notice until the story's snapshot came back empty. With AD-8, their onboarding checklist has one extra line: run tools/setup-hooks once. From that point on, their commits, checkouts, and Claude Code sessions are captured exactly like everyone else's, with no hook code to hand-copy or misconfigure.
TakeawayCapturing metrics only works if hooks are actually installed everywhere — so installing them is one committed, repeatable command, not something each developer sets up by hand.

Deliberately Not Decided Yet

What's Deferred — and Why

An architecture spine fixes only what needs fixing now to keep everything else consistent. These are intentionally left open:

Central presentation layer

Tech, hosting, and topology aren't chosen. Only its input contract — the versioned snapshot (AD-3) — is fixed. Any choice can plug in later without touching capture.

Snapshot schema versioning

We start at v1 with no migration tooling yet. Field-level shape and migration strategy will follow once real snapshots exist to learn from.

Manual-override path (AD-7)

No escape hatch yet if a developer ever works two stories on one branch. Branch-per-story is confirmed team convention today, so this is deferred, not ignored.

Credential provisioning (AD-4)

AD-4 fixes that credentials never enter shared files — but not how a developer's JIRA/Confluence token gets into their environment in the first place.

Story-point weight tuning (AD-6)

Phase-1/Phase-2 tables are seeded from an existing internal reference doc. Exact weights may need calibration once real usage data comes in.

TakeawayDeferring these doesn't block anything above — each one plugs in later without touching the capture pipeline we've already fixed.
Getting This Into Developers' Hands

How Would a Developer Actually Get This?

The architecture doesn't change based on how it's delivered — only the install experience does. Three realistic paths, roughly in build order:

Step 1 — today

Repo starter kit

Hooks, the setup script, and the kickoff/close skills ship inside the project repo (per AD-8). A developer clones the repo and runs tools/setup-hooks once — the same motion as npm install. No new tools to distribute.

Step 2 — next

Scaffolding CLI / template repo

An internal command (e.g. npx create-metrics-pipeline or an opsx subcommand) drops all of Step 1's files into any project, new or existing, and can re-run to pull updates — turns a one-off kit into something reusable across every team.

Step 3 — later

VS Code extension

A polished client on top of the same local files: auto-installs hooks on folder-open, shows a status-bar timer ("Story AIM-142 — 42m active"), and gives the kickoff prompt a native UI instead of a chat skill. It's a new face on the same substrate — not a replacement for it.

Why this order: Step 1 is buildable with everything already decided in this spine. Step 2 only wraps Step 1 in a reusable installer. Step 3 is the most valuable to end users but the most effort — and it never has to be built first for the metrics themselves to start flowing.
TakeawayStart by embedding this directly in the repo — it works immediately — then layer on a scaffolding tool and, eventually, a VS Code extension as adoption grows.
